## Q1: 如果一半失败，钱怎么算？

### 答：

> "失败的部分自动退款，  
> 最终只扣成功视频对应的 credits。  
> 
> 举个例子：  
> - 你提交 100 条 batch，预扣 1000 credits  
> - 成功了 60 条，失败了 40 条  
> - 最终扣 600 credits，退 400 credits  
> 
> 所有这些都是自动的，  
> 你可以在 Admin Dashboard 看到完整的账本记录。"

### 如果客户追问："退款多久到账？"

> "退款是实时的。  
> Worker 检测到失败后，  
> 立即调用 `finalize_batch_credits` RPC，  
> credits 立即回到你的钱包，  
> 可以继续使用。"

## Q2: 请求重试会不会重复扣费？

### 答：

> "不会。  
> 所有请求都有 `request_id` 幂等保护，  
> 同一个 `request_id` 只会生成一个 batch。  
> 
> 如果你因为网络问题重复发送了请求，  
> 系统会返回相同的 `batch_id`，  
> 不会重复扣费，不会重复创建任务。  
> 
> 这是数据库级别的唯一约束保证的，  
> 不是应用层的判断。"

### 如果客户追问："如果我的系统崩溃了，重试会怎样？"

> "完全安全。  
> 你可以在重试时使用相同的 `request_id`，  
> 系统会返回之前创建的 batch 信息，  
> 包括当前状态、成功数、失败数。  
> 
> 这是幂等设计的核心价值：  
> 网络抖动、系统崩溃、人工重试，  
> 都不会导致重复扣费。"

## Q3: 你们怎么防止 API 被滥用？

### 答：

> "我们有四层防护：  
> 
> **1. API Key 级限流**  
> 每个 API Key 都有 `rate_limit_per_min`，  
> 默认 60 次/分钟，可以自定义。  
> 
> **2. 分钟级 usage bucket**  
> 我们使用 `minute_bucket` 精确计数，  
> 数据库级别的硬限流，  
> 不会因为并发绕过限制。  
> 
> **3. 可绑定 IP（可选）**  
> 企业客户可以要求 IP 白名单，  
> 只有指定 IP 才能使用 API Key。  
> 
> **4. Admin 可随时禁用**  
> 如果发现异常，  
> Admin 可以立即禁用 API Key，  
> 不需要等客户响应。"

### 如果客户追问："如果我的 Key 泄露了怎么办？"

> "立即联系我们，  
> 我们可以在 1 分钟内禁用 Key。  
> 
> 而且，所有 API 调用都有 `request_id` 幂等保护，  
> 即使 Key 泄露，  
> 攻击者也无法通过重复请求造成重复扣费。  
> 
> 我们建议企业客户：  
> - 定期轮换 API Key  
> - 使用环境变量存储 Key  
> - 监控 API 使用情况"

## 反对意见处理技巧

### 1. 不要直接反驳
- ❌ "你说的不对"
- ✅ "我理解你的担心，让我解释一下"

### 2. 用数据说话
- ❌ "我们的系统很稳定"
- ✅ "我们的 SLA 是 99.9%，实际运行是 99.95%"

### 3. 用案例证明
- ❌ "很多客户都用我们的系统"
- ✅ "XX 公司（同行业）用我们的系统，每月生成 5000+ 视频"

### 4. 降低决策门槛
- ❌ "你可以试试"
- ✅ "我们跑一个 20 条视频的 Pilot，不收费，只看结果"

## 总结

**核心原则**:
1. ✅ 承认客户的担心（不要反驳）
2. ✅ 用数据/案例证明（不要空谈）
3. ✅ 降低决策门槛（推进 Pilot）

**关键话术**:
- "我理解你的担心"
- "让我用数据说明"
